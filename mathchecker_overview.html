MathChecker / Project Reading

它不是判答案对错，而是找出推理链里第一处错。

这个项目的核心任务是：给定一道数学题和模型生成的 多步解题过程， 系统要逐步检查每一步是否和题目条件、前序推理、算式变形相容， 最终定位出第一处错误步骤。

步级错误检测 第一处错误定位 数学推理审计 LLM + 工具 + 路由

一句话看项目定位

输入 题目 + 解题步骤轨迹
输出 每一步标签 + 首错位置
重点 不是生成答案，而是审查推理
难点 不同但正确的替代解法很多

项目更像什么

它更像一个“数学推理质检流水线”， 不是直接求解器。它把每个步骤拆开检查，再把 prompt judgment、 specialist tools、 review、 learned router 拼成一个多层审校系统。

这个项目核心是解决什么问题？

目标不是只看最终答案，而是对整条数学推理轨迹做定位式诊断。

定位首错，而不是只判终局。

MathChecker 的目标是判断一个多步解题过程里， 哪一步第一次出现了真正的数学冲突。 这类冲突可能来自算术错误、非法等价变形、错误代换、条件遗漏， 或者把尚未证明的分支当成最终结论。

它关注的是 “reasoning trace quality”，不是单个 final answer。

它为什么值得做

  • 只看最终答案，会漏掉“中间推理已经坏掉但碰巧答对”的情况。
  • 只看整段文本对错，无法告诉你模型是从哪一步开始偏离的。
  • 训练和评测推理模型时，首错位置比单纯答案正确率更有诊断价值。
  • 对教学、自动批改、推理系统调试都更有用。

为什么这件事难

数学轨迹里最大的陷阱不是“看起来不一样”，而是“看起来像错，但其实只是另一条正确路径”。

多种正确解法

同一道题可能有不同拆分方式、不同中间变量、不同等价写法。系统不能把“不同”直接当成“错误”。

局部变形真假难辨

很多错误不在自然语言里，而在局部算式：比如等价变形、代换、符号、括号、优先级、条件义务。

首错比终错更难

后续步骤往往会继承前面错误，所以系统不能只看某一步像不像错，还要找最早的那一步冲突来源。

它用了什么方法

核心思路是“分阶段判断 + 专家工具证据 + 可学习路由 + 复核纠偏”。

1

Stage 1：先生成软参考

先根据题目和前序步骤，抽取下一步应涉及的数学概念、关键分析和待计算表达式。 这一步不是直接判错，而是提供一个“应当关注什么”的软参考。

2

Stage 2：对当前步骤做三维局部判别

系统分别对 mathematical_concepts、 key_analyses、calculations 三个维度打标签；只要某一维出现 contradiction-found， 当前步就会被视为错误候选。

3

Hybrid Step Type Classifier：判断这一步“像什么动作”

用启发式规则 + LLM 分类 + 失败回退的方式，把步骤划成 decomposition、substitution、 algebraic_transformation、condition_case、 final_conclusion、arithmetic 等类型。

4

Specialist Router：决定要调哪些专项校验器

项目不让所有步骤都走同一套检查，而是按步骤类型和风险信号决定 应该调用哪类 specialist tool，并决定是否启动更重的 specialist review。

5

Specialist Tools：给出局部、结构化、可审计证据

三个主要工具分别检查：这是不是有效替代路径、这次代换/等价变形是否成立、 以及当前步骤是否满足题目或分支条件义务。

6

Review 层：把初判和工具证据重新整合

先做通用 stage2_review，再做带 specialist evidence 的 stage2_specialist_review，避免把“文本直觉”当成最终结论。

7

Deterministic Fallback：最后兜底纠偏

如果工具证据明确支持“其实是有效替代解法”，就把过于激进的矛盾判断放松； 如果工具给出硬冲突，而文本标签没发现，就把该步提升为真实矛盾。

关键方法拆解

从实现上看，这个项目不是单一算法，而是多个机制叠加后的推理审计流水线。

方法 1：两阶段 Prompt 结构

stage1 负责生成“下一步应关注的信息”， stage2 负责对当前步骤做局部判别。 这比一次性整段打分更稳定，也更方便缓存、复查和调试。

方法 2：三维标签，不只给一个 yes/no

项目把每一步拆成概念、分析、计算三个维度判断， 用 correct-and-aligned、 reasonable-but-incomplete、 nothing-extracted、 contradiction-found 来表达细粒度状态。

方法 3：工具证据进入复核链路

不是让工具直接“一票否决”，而是把工具结果作为结构化 evidence 送入 review 层，再与原始 Stage 2 结果整合。

三个 specialist tool 在干什么

它们分别处理最容易造成误判的三类局部问题。

alternative_route_verifier_tool

解决“和参考路径不一样，但其实仍然正确”的问题。 这个工具本质上在给系统加容错，避免把替代解法误判为首错。

equivalence_substitution_verifier_tool

处理代换、改写、分解、等价变形是否成立。 它针对的是真正的数学结构问题，而不是表面措辞差异。

condition_obligation_verifier_tool

检查当前步骤有没有违反题目条件、分支条件或结论义务， 比如少讨论情况、把临时假设偷换成结论等。

Learned Router 用了什么学习方法

这个模块不是生成文本，而是学习“这一类步骤该调哪些 specialist”。

输入表示

路由输入包含题目、前序步骤、当前步骤、启发式步骤类型、风险标志， 以及 Stage 1 产出的概念/分析/计算提示。

训练目标

用多标签分类预测四件事： 是否调用替代路径 verifier、等价变形 verifier、条件义务 verifier， 以及是否触发 specialist review。

模型路线

仓库里给出的训练路线是 Qwen/Qwen3-0.6B + sequence classification + multi-label routing + LoRA 微调， 并支持低置信度时回退到规则路由。

MathChecker 的核心方法可以概括成一句话： 用分阶段判断缩小问题，用 specialist 工具提供硬证据，用 learned router 让“该调谁”这件事也可学习。

所以它既不是纯 prompt 方案，也不是纯符号系统，而是一个 “LLM 判断 + 局部验证器 + 训练型路由器 + 多轮复核” 的混合架构。

我对这个项目的简化理解

如果要用产品语言来总结，它更像一个“数学推理过程审计器”。

它最想解决的痛点

让系统不要只会说“这题答错了”，而是能进一步说出： “你从第几步开始错了，错在什么类型的局部冲突上， 以及这个判断背后有没有结构化证据支持。”

它最关键的方法选择

  1. 把问题拆到“单步”粒度，而不是整段一起判。
  2. 把“步骤类型识别”和“specialist 路由”拆成两层。
  3. 允许 learned router 学会“何时调用哪种 verifier”。
  4. 保留 review 和 deterministic fallback，增强稳定性。
